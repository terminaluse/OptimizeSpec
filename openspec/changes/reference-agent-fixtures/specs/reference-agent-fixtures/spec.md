## ADDED Requirements

### Requirement: Reference agents are committed as input fixtures
The system SHALL manage reference agents as hand-authored test inputs rather than as complete committed optimization systems.

#### Scenario: Reference agent fixture exists
- **WHEN** a test or skill needs a representative target agent
- **THEN** it loads a fixture from a dedicated reference-agent fixture location containing source inputs such as `agent.yaml`, `request.md`, and minimal fixture metadata

#### Scenario: Fixture is reviewed
- **WHEN** a reference-agent fixture is reviewed in the repository
- **THEN** it does not include generated runner files, optimizer traces, run ledgers, or full generated `optimizespec/changes/<change-id>` systems

### Requirement: Optimization systems are generated during tests
The system SHALL generate optimization-system artifacts during CLI, skill, and harness tests instead of relying on prebuilt committed optimization-system directories.

#### Scenario: CLI generation is tested
- **WHEN** a test verifies that the CLI can apply an optimization-system spec
- **THEN** the test creates a temporary target workspace and asserts that generated files are written there

#### Scenario: Skill workflow is tested
- **WHEN** a test verifies that skills can produce an optimization system from a reference agent
- **THEN** the generated proposal, design, specs, tasks, runners, and evidence outputs are created in a temporary or ignored workspace

#### Scenario: Generated output is accidentally committed
- **WHEN** release-visible repository paths contain generated `optimizespec/changes/*`, generated optimization-system folders, run ledgers, or optimizer traces that are not explicitly approved narrow snapshots
- **THEN** verification fails with guidance to move the data to a temporary run path, ignored local output, or focused expected-output fixture

### Requirement: Expected-output fixtures stay narrow
The system SHALL keep deterministic expected-output fixtures focused on stable contracts and SHALL NOT treat them as runnable optimization systems.

#### Scenario: Snapshot is useful
- **WHEN** a deterministic comparison needs a golden fixture
- **THEN** the fixture contains only the stable contract surface being asserted, such as CLI JSON output, manifest shape, or a short generated file excerpt

#### Scenario: Snapshot grows into a system
- **WHEN** an expected-output fixture becomes a full runnable optimization system with generated scripts, eval data, and run instructions
- **THEN** maintainers either convert it back to a focused snapshot or generate it during the relevant test

### Requirement: Python reference harness remains non-product
The system SHALL keep any Python Managed Agents harness code clearly separated from product CLI behavior and generated optimization-system artifacts.

#### Scenario: Python harness is used by regression tests
- **WHEN** deterministic or live tests use the Python Managed Agents reference implementation
- **THEN** they treat it as a test harness or reference runtime and do not require committed generated optimization-system changes as canonical source

#### Scenario: Package is inspected
- **WHEN** maintainers run release packaging verification
- **THEN** the package excludes reference-agent fixtures, Python harness examples, generated optimization systems, run outputs, and Python package metadata unless a file is intentionally allowlisted as a runtime template

### Requirement: Documentation explains fixture ownership
The system SHALL document where reference agents, generated outputs, expected snapshots, and local run artifacts belong.

#### Scenario: Contributor adds a new reference agent
- **WHEN** a contributor wants to add a representative agent for testing
- **THEN** documentation directs them to add a reference-agent fixture and a test that generates optimization-system outputs from it

#### Scenario: Contributor adds generated artifacts
- **WHEN** a contributor attempts to add generated optimization-system artifacts to a committed path
- **THEN** documentation and verification explain why those artifacts should be generated by tests or kept in ignored local output
